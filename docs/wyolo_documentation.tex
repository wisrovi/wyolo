\chapter{Executive Summary}
\cleardoublepage
The \textbf{wyolo} project represents a paradigm shift in how computer vision models, specifically YOLO (You Only Look Once) and RT-DETR architectures, are trained and managed in enterprise environments. Traditionally, deep learning training relies on monolithic scripts that are prone to silent failures, resource leakage, and lack of traceability. 

This document outlines the architecture, implementation, and operational guidelines for \textbf{wyolo}, a highly resilient, state-driven worker microservice. By wrapping the core Ultralytics engine within the \texttt{wpipe} state machine orchestration library, the system ensures that GPU resources are only engaged when all external dependencies (MinIO/S3, MLflow, and datasets) are rigorously validated.

Furthermore, \textbf{wyolo} provides native MLOps integration, automating the logging of hyperparameters, system metrics, and artifact synchronization. This documentation serves as a comprehensive guide for Software Architects, Data Scientists, and DevOps Engineers to deploy, configure, and maintain the wyolo framework in production environments.

\chapter{Introduction}
\cleardoublepage
\section{Background}
In the domain of Computer Vision, the transition from experimental Jupyter notebooks to production-grade training pipelines is fraught with challenges. Data drift, out-of-memory (OOM) errors, and the loss of model lineage are common pitfalls. The \textbf{wyolo} library was conceived to address these exact challenges by introducing software engineering best practices---such as state machines, dependency injection, and automated telemetry---into the deep learning lifecycle.

\section{Objectives}
The primary objectives of the \textbf{wyolo} architecture are:
\begin{itemize}
    \item \textbf{Resiliency:} Implement a "Fail Fast, Fail Cheap" mechanism using a pre-training validation state machine.
    \item \textbf{Traceability:} Guarantee 100\% reproducibility by forcing integration with DVC (Data Version Control) and MLflow.
    \item \textbf{Scalability:} Decouple the training logic into a stateless worker container that can be scaled horizontally via Kubernetes or Docker Swarm.
    \item \textbf{Automation:} Remove the need for manual hyperparameter tuning and model registry management.
\end{itemize}

\chapter{Technical Architecture}
\cleardoublepage
\section{System Overview}
The system is divided into three distinct layers: the Orchestration Layer, the Domain Logic Layer, and the Engine Wrapper Layer.

\subsection{Orchestration Layer}
Powered by \texttt{wpipe}, this layer manages the Directed Acyclic Graph (DAG) of the training lifecycle. It handles timeouts, retries, and conditional routing. The pipeline is defined in \texttt{app/main.py}.

\subsection{Domain Logic Layer}
This layer consists of atomic states located in \texttt{app/states/}. Each state is responsible for a single validation or execution step. For example, \texttt{check\_gpu\_available} verifies CUDA availability before any PyTorch tensors are allocated.

\subsection{Engine Wrapper Layer}
The \texttt{TrainerWrapper} class in \texttt{core/trainer\_wrapper.py} abstracts the Ultralytics API. It intercepts callbacks to inject MLflow logging and MinIO artifact synchronization seamlessly.

\section{State Machine Workflow}
The following steps detail the deterministic pipeline execution:
\begin{enumerate}
    \item \textbf{load\_yaml}: Parses the configuration payload.
    \item \textbf{check\_minio\_buckets}: Validates S3 connectivity and credentials.
    \item \textbf{check\_gpu\_available}: Probes PCIe for CUDA devices and calculates available VRAM.
    \item \textbf{check\_dataset}: Verifies the structural integrity of the input dataset.
    \item \textbf{train\_model}: Executes the YOLO training loop if and only if all prior checks pass.
    \item \textbf{error\_capture}: Acts as the global exception trap, ensuring the worker exits gracefully and reports telemetry on failure.
\end{enumerate}

\chapter{Installation Guide}
\cleardoublepage
\section{Prerequisites}
Ensure the following components are available in your host environment:
\begin{itemize}
    \item Python 3.8 or higher.
    \item NVIDIA GPU with CUDA 11.8+ installed.
    \item Docker and NVIDIA Container Toolkit (for containerized execution).
\end{itemize}

\section{Local Installation}
To install the library for local development:

\begin{lstlisting}[language=bash, caption=Local Installation Commands]

git clone https://github.com/wisrovi/wyolo.git
cd wyolo

# Create virtual environment
python3 -m venv venv
source venv/bin/activate

# Install with development dependencies
make install
\end{lstlisting}

\section{Docker Deployment}
For production environments, deploying via Docker is highly recommended. The included \texttt{Dockerfile} handles the installation of system-level graphics libraries required by OpenCV and Ultralytics.

\begin{lstlisting}[language=bash, caption=Docker Build and Run]
# Build the image
docker build -t wyolo-worker:latest -f src/wyolo/Dockerfile .

# Run the container with GPU access
docker run --gpus all \
    -v $(pwd)/configs:/workspace/configs \
    wyolo-worker:latest \
    wyolo-train --config_path /workspace/configs/config.yaml
\end{lstlisting}

\section{Core Trainer Implementation}
The \texttt{TrainerWrapper} class intercepts the training lifecycle to inject telemetry.

\begin{lstlisting}[language=python, caption=TrainerWrapper Implementation Excerpt]
import mlflow
from slugify import slugify
from ultralytics import YOLO

class TrainerWrapper:
    def __init__(self, config: dict):
        self.config = config
        
    def on_train_end(self, trainer):
        if "mlflow" in self.config:
            pytorch_model = trainer.model.model
            registered_model_name = f"{self.config['sweeper']['study_name']}"
            
            # Log model to MLflow Model Registry
            mlflow.pytorch.log_model(
                pytorch_model=pytorch_model,
                artifact_path="model",
                registered_model_name=registered_model_name
            )
            
            # Sanitize and log metrics
            metrics = {slugify(k): float(v) for k, v in trainer.metrics.items()}
            mlflow.log_metrics(metrics)
\end{lstlisting}

\chapter{Practical Use Cases}
\cleardoublepage
\section{Automated Hyperparameter Sweeps}
In conjunction with Ray Tune or Optuna, \textbf{wyolo} can act as the execution worker. The orchestrator spawns multiple containers, injecting different YAML configurations. Because \textbf{wyolo} handles its own state and artifact tracking via MLflow, the orchestrator only needs to monitor the final status.

\section{Continuous Training (CT) Pipelines}
When new data is annotated and merged into the DVC repository, a CI/CD pipeline (e.g., GitHub Actions or GitLab CI) triggers a \textbf{wyolo} worker. The worker verifies the new dataset footprint using \texttt{check\_dataset.py}, runs the training, and registers the new champion model automatically.

\chapter{Best Practices and Troubleshooting}
\cleardoublepage
\section{Best Practices}
\begin{itemize}
    \item \textbf{Use Autobatching:} Always set \texttt{batch: -1} in production configurations unless you have a strict deterministic requirement.
    \item \textbf{Sanitize Dataset Names:} Ensure your DVC paths and YAML configurations do not contain spaces or special characters to prevent MLflow URI parsing errors.
    \item \textbf{Monitor SQLite WAL:} The \texttt{wtrain.db} file provides an immutable audit trail. Ingest this database into your monitoring stack for fleet-wide visibility.
\end{itemize}

\section{Troubleshooting Guide}
\begin{itemize}
    \item \textbf{Issue:} Pipeline halts at \texttt{check\_minio\_buckets}.
    \item \textbf{Resolution:} Verify the \texttt{endpoint} string. Do not include \texttt{http://} if the SDK expects raw domains. Ensure the container has network egress to the S3 port.
    \item \textbf{Issue:} CUDA Out of Memory during \texttt{train\_model}.
    \item \textbf{Resolution:} If autobatch fails, manually set \texttt{batch: 16} and increase the \texttt{retry\_on\_exceptions} parameter in the \texttt{wpipe} orchestrator to attempt a fallback resolution.
\end{itemize}

\chapter{Conclusions and Future Work}
\cleardoublepage
\section{Conclusion}
The \textbf{wyolo} framework successfully bridges the gap between raw computer vision research and enterprise MLOps. By enforcing a state-machine driven lifecycle, it significantly reduces the operational overhead of training YOLO architectures at scale.

\section{Future Work}
Future iterations of \textbf{wyolo} will focus on:
\begin{itemize}
    \item Native integration with Kubernetes Custom Resource Definitions (CRDs) for spawning distributed multi-node training jobs via \texttt{torchrun}.
    \item Expanded support for emerging architectures beyond Ultralytics (e.g., HuggingFace Transformers).
    \item Advanced dataset caching mechanisms using Redis to accelerate data loading in ephemeral containers.
\end{itemize}
